Deine eigene Produkt-Surface

Heute habt ihr eine Shader-Surface in den Lumen-Hero gebaut, gesteuert über uColor und uTime. Jetzt baut ihr eure eigene Oberfläche für eine echte App-Situation und deployt sie live.

zero-build · eine index.html three@0.184.0 · Importmap CPU → uniforms → GPU

Am Ende habt ihr eine URL, die ihr verschicken könnt, und eine kurze Engineer-Note.

Das Ergebnis ist kein Shader-Bild auf einem leeren Canvas. Es ist eine Oberfläche in einem glaubwürdigen App-Kontext, mit etwas Produkt-Text drumherum, damit klar wird: der Shader dient der App.

Nimm deinen Kurs-Stand als Vorlage

Schritt 1 · App-Situation

Wähle genau eine Situation.

Sie entscheidet über Stimmung, Farbe und Bewegung deiner Surface. Es gibt keine bessere oder schlechtere Wahl. Nimm die, die dich reizt.

01Auth

Auth-Hintergrund

Ein Login- oder Signup-Screen, die Surface liegt hinter dem Formular.

Stimmungruhig, tief, einladend
02Loading

Loading-Shimmer

Ein Skeleton-Zustand, der lädt, statt zwanzig DOM-Knoten pulsen zu lassen.

Stimmungsanfte, langsame Bewegung
03Empty

Empty-State

„Noch keine Daten“, eine Fläche, die lebt, statt leer zu wirken.

Stimmungkühl, geduldig
04CTA

CTA-Fläche

Ein Call-to-Action-Block, der auf Hover oder Klick reagiert.

Stimmungwarm, einladend, ein Impuls
05Onboarding

Onboarding-Hero

Ein Willkommens-Screen mit einem großen Titel.

Stimmunghell, freundlich, offen

Schritt 2 bis 4 · Bauen, erklären, deployen

Von der Plane zur Live-URL.

Drei Schritte in Folge. Mehr Geometrie als eine Plane braucht ihr nicht.

02
Bau die Surface

Die Surface lebt in einem App-Kontext

  • Die Surface lebt in einem App-Kontext: Produkt-Text, ein Titel oder ein Formular davor oder daneben, nicht nur ein nacktes Canvas.
  • Ein varying reicht vUv durch, der Fragment-Shader baut daraus einen Verlauf mit mix.
  • Mindestens ein eigenes Farb-uniform (uColor oder ein sinnvollerer Name für eure Situation), gesteuert aus dem JavaScript.
  • uTime bewegt die Fläche ruhig.Eine Zahl pro Frame, der Rest passiert auf der GPU.
  • Resize-korrekt: die Surface füllt ihre Fläche auch nach Größenänderung sauber.
  • Sauberer Code: die uniform-Namen stimmen auf beiden Seiten Zeichen für Zeichen.
03
Die Engineer-Note

Drei Fragen, in der README beantwortet

Legt eine README.md ins Repo und beantwortet darin drei Fragen in je zwei, drei Sätzen. Das ist der Teil, der euch von jemandem unterscheidet, der Code kopiert. Schreibt es in euren eigenen Worten.

  • Was injiziert ShaderMaterial für euch, das ihr nirgends deklariert habt?
  • Was lässt RawShaderMaterial offen, also was müsstet ihr selbst schreiben?
  • Worüber reden Vertex- und Fragment-Shader miteinander, und was passiert, wenn ein Name nicht zusammenpasst?
04
Deploy live

Bonus · frei, wenn ihr Lust habt

Dieselbe Surface als RawShaderMaterial

Baut dieselbe Surface zusätzlich einmal als RawShaderMaterial, mit allen selbst deklarierten Zeilen (precision, die Matrizen, die Attribute).

Notiert in der README einen Satz, wie viele Zeilen Code euch das gegenüber ShaderMaterial zusätzlich gekostet hat. Das ist Tesler’s Law in einer Zahl: die Komplexität verschwindet nicht, sie wandert nur von der Bibliothek in euren Code.

Vor der Abgabe

Selbstcheck.

Sieben Punkte, die alle stimmen sollten, bevor ihr die URL verschickt.

  • Die Surface liegt in einem glaubwürdigen App-Kontext, nicht auf einem leeren Canvas.
  • Ein varying reicht vUv durch, der Verlauf entsteht mit mix.
  • Ein eigenes Farb-uniform wird aus dem JavaScript gesteuert.
  • uTime bewegt die Fläche, die CPU setzt nur eine Zahl pro Frame.
  • Die Seite ist resize-korrekt und hat keine Fehler in der Konsole.
  • Die README.md beantwortet die drei Fragen der Engineer-Note.
  • Die Live-URL läuft in einem frischen Browserfenster.
